OCR 表单提取系统 V1 实施方案

本项目要做的是一个面向表单类 PDF / 图片文件的半自动 OCR 提取系统。
目标不是做整页文字识别展示，而是做成一个可投入实际业务使用的文档结构化提取工具：

自动预选 + 手动微调 + OCR 预输入 + 模板复用 + 表格导出

文档的定位是 V1 可实施方案，既保留原始产品目标，也补齐实现所需的关键决策，确保研发团队拿到后可以直接进入设计和编码阶段。

一、项目定位

系统定位为半自动、人工确认型的表单提取工具，适用于版式相对固定、字段明确、重复处理频率较高的业务文档。

V1 的核心价值不是追求“全自动识别一切文档”，而是解决以下问题：

用户虽然识别出了文字，但提取不到真正想要的字段
用户看不到识别内容在原图上的来源位置
用户不能方便地修正 OCR 结果
同类表单重复处理时，无法复用历史配置
英文与数字混合字段容易出错，例如 Container No.、编号、日期、电话等

V1 的总体原则如下：

第一，固定字段优先，不在第一版追求通用任意文档识别
第二，半自动优先，允许人工确认和修正，不追求完全无人值守
第三，可复用优先，所有人工框选和字段定义应可沉淀为模板
第四，可部署优先，方案必须适配腾讯云单机 4 核 4GB 的资源边界

二、V1 范围与非目标

1. 输入范围

支持：

PDF
JPG
PNG

支持多页 PDF 上传，但 V1 采用分页处理模式。用户可以逐页预览和提取，模板按单页保存和应用，不做整份复杂 PDF 的自动模板分类。

2. 输出范围

V1 输出统一为结构化表格数据，主要形态为：

后端返回 Excel 友好的 JSON 数据结构
前端支持下载 CSV

V1 不直接生成复杂 `.xlsx` 文件，`.xlsx` 文件导出放到第二阶段。

3. V1 必做功能

文件上传
PDF 分页与页面预览
自动文本区域检测
候选框展示
区域框与提取框编辑
提取框区域 OCR
OCR 结果预填到右侧字段表单
字段手工修改
模板保存与复用
基础格式校验
低置信度提示
结构化导出

4. V1 明确不做

Word 文档解析
Excel 文档直接解析
通用任意版式识别
复杂多行表格自动抽取
复杂权限系统
多项目管理
全自动无人工确认流程
高并发批处理流水线

三、用户流程

V1 的完整用户流程如下：

用户上传 PDF 或图片
系统对文件进行基础校验并生成页面预览
系统对当前页面执行质量检查
系统执行文字区域检测并生成候选框
用户在左侧页面上调整区域框和提取框
系统仅对提取框区域执行 OCR
系统将识别结果预填到右侧字段表单
系统执行字段规则校验并给出风险等级
用户人工修改、确认结果
用户可选择保存为模板
用户导出当前页或多页合并后的结构化结果

该流程强调“机器先做粗识别，用户做低成本修正”，确保 V1 在识别精度不完美的情况下仍然可用。

四、系统架构

V1 在现有仓库基础上扩展，不新开独立系统。推荐架构如下：

1. React 前端

继续使用现有 React 前端，新增 OCR 工作台页面。

前端负责：

文件上传
PDF 页面预览
候选框、区域框、提取框渲染
拖动、缩放、删除、命名字段
右侧结果表单与风险提示
模板保存与套用
导出结果

2. Node/Express 主后端

继续使用现有 Node/Express 服务，负责业务编排和持久化。

主后端负责：

上传文件管理
PDF 转页调度
模板 CRUD
提取任务创建与状态管理
OCR 服务调用
结果存储
导出接口

3. Python OCR Service

新增独立 Python 服务，专门运行 PaddleOCR。

OCR 服务负责：

图片质量检查
文字检测
提取框区域识别
输出文本、坐标、置信度

V1 不使用 Tesseract 作为双引擎，也不将 LLM 作为主识别链路，以降低复杂度和运行成本。

4. PostgreSQL

继续使用 PostgreSQL 持久化模板、文档、任务、识别结果和人工修订记录。

5. 任务执行模式

V1 使用数据库任务表加单 worker 串行处理方式，不强制引入 Redis 或复杂队列系统。

这样做的原因是：

更适合腾讯云 4 核 4GB 的资源边界
实现成本更低
更符合小团队低到中低并发场景

五、OCR 技术路线

1. 主引擎选择

V1 主 OCR 引擎固定为 PaddleOCR。

选择原因：

对中英文、数字、表单场景的综合表现更稳
支持文字检测和识别的完整链路
便于后续扩展字段纠错和方向分类

V1 不采用以下方案作为主路线：

Tesseract 双引擎切换
多模态大模型直接读整页出结构化结果
OCR 结果完全依赖外部按 token 计费的 AI 服务

因此，V1 主链路不依赖按 token 计费的 AI 服务，运行成本主要来自腾讯云服务器资源，而不是 API 调用费用。

2. OCR 执行策略

V1 不做整页识别后再强行猜字段，而是采用两段式识别：

先对页面做候选框检测
再对用户确认后的提取框区域做精确 OCR

这样可以减少无关文本干扰，提升字段识别稳定性，也更符合“可解释”和“可修正”的产品目标。

六、框选机制与交互原则

系统保留两类框：

1. 区域框

用于划分页面大的语义区域，例如：

Header Area
Table Area
Footer Area
Signature Area

区域框的主要作用：

帮助系统理解页面结构
限制候选框搜索范围
降低表格线、邻近文本和噪声干扰

2. 提取框

用于定义具体要提取的字段，例如：

DN No.
Container No.
BL No.
ETD
ETA
Truck No.
Phone

提取框的主要作用：

绑定字段名和输出列
作为 OCR 裁剪区域
支撑模板复用
支撑风险标记和人工修正

七、模板策略

模板是 V1 的核心能力，必须支持保存、加载和复用。

1. 模板保存内容

每个模板至少保存以下信息：

模板名称
模板说明
适用页尺寸
创建时间
更新时间
区域框列表
提取框列表
字段定义
输出列名
字段类型
校验规则
是否必填
风险提示规则
锚点配置

2. 坐标保存策略

模板中的框坐标必须保存为归一化坐标，而不是固定像素坐标。

归一化坐标的意义是：

同一类表单在不同分辨率、不同缩放比例下仍可复用
为轻量模板偏移适配提供基础

3. 锚点机制

为了让模板在“同类单据存在轻微偏移”时仍可使用，模板应支持锚点信息。

锚点可以是：

参考文本
参考小区域
关键字段附近的稳定位置

4. 偏移适配策略

V1 的模板偏移适配只支持轻量修正，不做复杂图像配准。

V1 允许的自动适配包括：

基于页面宽高的缩放映射
基于锚点位置的平移修正
小范围缩放偏差修正

V1 不支持：

大角度旋转后的复杂自动对齐
跨版式模板自动泛化
严重错位页面的强制自动套用

如果模板锚点缺失、偏差过大或匹配不稳定，系统必须提示用户手动微调，而不是静默套错模板。

九、API 与服务边界

1. 前端对主后端接口

建议提供以下接口：

POST /ocr/documents
上传文件并创建文档记录

GET /ocr/documents/:id/pages
获取文档分页信息与预览数据

POST /ocr/pages/:pageId/detect
对单页执行文字候选框检测

POST /ocr/pages/:pageId/extract
对当前提取框列表执行 OCR

GET /ocr/jobs/:id
查询任务状态与处理结果

POST /ocr/templates
保存模板

GET /ocr/templates
查询模板列表

GET /ocr/templates/:id
查询模板详情

POST /ocr/templates/:id/apply
将模板应用到指定页面

POST /ocr/export
导出结构化结果或 CSV

2. Node 后端与 OCR Service 内部接口

建议提供以下内部接口：

POST /quality-check
输入页面图像，返回质量评分与问题列表

POST /detect
输入页面图像，返回候选框列表和检测置信度

POST /recognize
输入页面图像和提取框列表，返回每个框的识别文本和置信度

3. 服务职责边界

边界应明确如下：

Node 后端负责业务语义、模板、导出、任务状态和数据存储
Python OCR Service 负责图像算法、检测和识别
前端不直接调用 OCR Service，只通过主后端访问

十、前端交互要求

V1 的核心交互保留左图右表单模式。

1. 左侧页面区域

支持：

显示原始图片或 PDF 页面
显示候选框
显示区域框和提取框
新增框
删除框
拖动和缩放框
高亮选中框

2. 右侧结果表单

支持：

显示字段名
显示 OCR 原始结果
显示最终结果
显示风险状态
允许用户直接修改
支持新增字段
支持修改输出列名和字段类型

3. 联动交互

要求实现：

点击左侧框，右侧对应字段高亮
点击右侧字段，左侧对应框高亮
OCR 结果自动填入字段输入框
字段修改后自动标记为人工编辑
模板应用后如有偏移异常，明确提示用户复核

十一、校验与风险提示

V1 必须在 OCR 后增加规则校验层，而不是仅展示识别文本。

1. 风险等级

风险等级固定为三档：

normal
review
high_risk

2. 风险来源

风险等级由以下因素综合决定：

OCR 置信度偏低
字段格式校验失败
提取框内容可能截断
模板偏移匹配不稳定

3. V1 支持的字段类型与规则方向

text
自由文本，不做强约束

number
校验纯数字或小数格式

date
校验常见日期格式，并在可能情况下做标准化

phone
校验电话或手机号格式

code
校验字母数字组合，可允许自定义正则

container_no
预留集装箱编号格式规则，识别后重点检查 O/0、I/1、B/8、S/5 等易错字符

4. 结果保留要求

每个字段结果都必须保留：

OCR 原始值
校验状态
失败原因
人工修订值
是否已人工确认

十二、部署方案与资源约束

1. 腾讯云目标配置

目标服务器配置为：

CPU 4 核
内存 4GB
系统盘 40GB SSD

该配置可支撑 V1 上线试运行，但前提是范围控制在小团队低到中低并发场景。

2. 适用规模

建议适用规模：

1 到 10 人小团队
白天有零散上传和提取请求
同一时刻 1 到 2 个 OCR 任务
用户愿意人工确认结果

3. 部署建议

建议部署组件：

React 前端
Node/Express 主后端
Python OCR Service
PostgreSQL

建议 OCR worker 数量固定为 1，避免 CPU 和内存争抢导致整机卡死。

4. 文件与磁盘策略

40GB 磁盘对 V1 够用，但必须有清理策略。

建议规则：

上传原件保留固定周期
PDF 渲染出的中间页图在任务完成后按策略清理
OCR 裁剪临时文件任务完成即删
导出文件不长期缓存

5. PDF 页数与处理边界

V1 建议限制：

单次 PDF 页数上限，例如 10 页以内
单页渲染 DPI 控制在适合 OCR 的区间，例如 150 到 200 DPI
超过限制时给出明确提示，而不是继续硬处理

十三、验收标准

V1 完成后，应满足以下验收要求：

1. 功能验收

用户可以上传 PDF 或图片表单
系统可以完成 PDF 分页和页面预览
系统可以自动给出候选文字框
用户可以手动调整区域框和提取框
用户可以为提取框命名字段并设置字段类型
系统可以对提取框区域执行 OCR 并预填结果
用户可以修改 OCR 结果并保存修订值
用户可以保存模板并在相似页面中复用
系统可以对低置信字段做风险提示
最终结果可以输出为结构化表格数据和 CSV

2. 性能验收

在腾讯云 4 核 4GB 环境下：

单页普通表单的目标处理时间不高于 10 秒
系统可稳定支持同时 1 到 2 个 OCR 任务
OCR 任务失败时必须返回明确状态，不允许静默失败

3. 模板验收

单页模板可以成功保存与加载
同类表单在轻微偏移时可成功复用模板
偏移超过阈值时系统会提示手动微调，而不是错误提取

4. 风险提示验收

日期、电话、编号、Container No. 等高风险字段有明显标识
校验失败原因可展示给用户
人工修订后结果可覆盖最终导出值

十四、第二阶段规划

以下能力不纳入 V1，但应在设计时预留扩展空间：

多行表格区域自动抽取
原生 `.xlsx` 文件导出
更强的模板匹配与模板自动推荐
复杂权限体系和多人协作
批量任务与高并发队列
引入 LLM 做低置信字段纠错、字段归一化和模板辅助推荐

十五、实施原则

研发执行时应遵循以下原则：

前后端解耦
OCR 能力与业务编排解耦
模块职责清晰
先实现稳定可用的 V1，再考虑复杂智能化能力
所有高影响决策都应围绕“固定字段优先、单页模板优先、无 LLM 主链路、适配 4 核 4GB 单机部署”展开

本实施方案确认后，可直接据此进入数据库设计、接口设计、前端页面拆分和 OCR 服务开发阶段。
